# 문자 인코딩 *

컴퓨터에서 문자도 이진수로 저장됩니다. 문자와 숫자를 서로 바꾸려면 각 문자에 고유한 숫자 코드를 대응시키는 **문자 집합**이 필요합니다.

## ASCII 문자 집합

ASCII는 American Standard Code for Information Interchange의 약자로 가장 오래된 문자 집합 가운데 하나입니다. ASCII는 7비트로 최대 128개의 문자를 표현합니다. 영문자, 숫자, 문장 부호와 줄바꿈·탭 같은 제어 문자가 포함됩니다.

![ASCII 코드표](character_encoding.assets/ascii_table.png)

ASCII는 영어만 지원합니다. EASCII는 8비트, 256개 코드로 범위를 넓혔습니다. 여러 EASCII 변형은 앞 128개 코드를 ASCII와 같게 유지하고 나머지 128개를 지역의 필요에 맞게 정의합니다. 그러나 256개 코드만으로는 많은 문자 체계를 담을 수 없습니다.

## GBK 문자 집합

1980년 중국어 컴퓨터 처리를 위해 6,763개의 한자를 포함한 GB2312 표준이 발표되었습니다. 이후 GBK가 GB2312를 확장하여 희귀 한자와 번체자를 포함한 총 21,886개의 한자를 지원했습니다. GBK 인코딩에서는 ASCII 문자는 1바이트, 한자는 보통 2바이트를 사용합니다.

GBK는 언어별 문자 집합이 지역의 요구는 해결할 수 있지만 다국어 환경에서는 불편하다는 점을 보여 줍니다.

## Unicode 문자 집합

여러 문자 집합이 공존하면 서로 다른 표준을 사용하는 시스템 사이에서 데이터가 깨진 문자로 보일 수 있습니다. Unicode는 전 세계의 문자와 기호에 하나의 통합된 코드 공간을 제공하기 위해 만들어졌습니다.

Unicode는 각 문자에 `U+0000`부터 `U+10FFFF` 사이의 고유한 **코드 포인트**를 할당합니다. 다만 Unicode는 문자와 코드 포인트를 정의할 뿐 바이트를 저장하는 단 하나의 방식까지 정하지는 않습니다. 실제 저장에는 UTF-8, UTF-16, UTF-32 같은 인코딩 형식이 필요합니다.

가장 단순한 방법은 모든 문자에 같은 바이트 수를 할당하는 것입니다. 예를 들어 문자가 항상 2바이트라면 문자열을 나누기는 쉽지만 ASCII 문자만 있는 텍스트도 필요한 공간의 두 배를 사용합니다.

![Unicode 코드 포인트 표현 예](character_encoding.assets/unicode_hello_algo.png)

## UTF-8 인코딩

UTF-8은 널리 사용되는 Unicode 인코딩입니다. 한 문자를 1바이트부터 4바이트까지로 표현하는 **가변 길이 인코딩**입니다.

- ASCII 문자는 1바이트를 사용합니다. 최상위 비트는 $0$이고 나머지 일곱 비트에 코드 포인트를 담습니다. 따라서 UTF-8은 ASCII와 하위 호환됩니다.
- 문자가 $n$바이트라면 첫 바이트 앞의 $n$개 비트가 $1$이고 다음 비트가 $0$입니다. 이어지는 바이트는 각각 `10`으로 시작하며 남은 비트에 코드 포인트를 나누어 담습니다.

첫 바이트의 연속된 `1` 개수를 통해 한 문자의 바이트 길이를 알 수 있습니다. 후속 바이트의 `10` 접두사는 읽기가 문자 중간에서 시작되었는지도 판단하게 해 줍니다.

![UTF-8 인코딩 예](character_encoding.assets/utf-8_hello_algo.png)

Unicode의 다른 대표적인 인코딩은 다음과 같습니다.

- **UTF-16**: 문자 하나에 2바이트 또는 4바이트를 사용합니다. 자주 쓰는 많은 문자는 2바이트이고 기본 다국어 평면 밖의 문자는 4바이트 서로게이트 쌍이 필요합니다.
- **UTF-32**: 코드 포인트마다 고정된 4바이트를 사용합니다. 위치 계산은 쉽지만 보통 더 많은 공간이 필요합니다.

UTF-8은 ASCII 문자가 많은 텍스트에서 특히 효율적이며 호환성이 뛰어납니다. UTF-16은 일부 비ASCII 문자 집합에서 더 적은 공간을 사용할 수 있습니다. 적절한 선택은 데이터와 처리 환경에 따라 달라집니다.

## 프로그래밍 언어의 문자 인코딩

문자열이 메모리에 저장되는 방식은 문자 접근과 길이 계산에 영향을 줍니다.

- 고정 길이 단위 표현에서는 위치 접근과 길이 계산이 비교적 단순할 수 있습니다.
- 가변 길이 UTF-8에서는 문자 하나가 여러 바이트일 수 있으므로 $i$번째 문자를 찾으려면 문자열 앞에서부터 읽어야 할 수 있습니다.
- 자르기, 이어 붙이기, 삽입, 삭제는 코드 포인트 경계를 지켜야 잘못된 UTF-8 바이트열을 만들지 않습니다.

프로그래밍 언어마다 구현 선택도 다릅니다.

- Java, JavaScript, TypeScript, C#은 문자열에 UTF-16 코드 단위를 사용합니다. 16비트 범위를 벗어나는 문자는 서로게이트 쌍으로 표현합니다.
- CPython은 보통 유연한 문자열 표현을 사용하며 단위당 크기는 문자열에 등장하는 가장 큰 코드 포인트에 따라 달라집니다. 이는 구현 세부 사항이지 모든 Python 인터프리터의 공통 보장은 아닙니다.
- Go는 `string`을 UTF-8 바이트로 저장하고 Unicode 코드 포인트 하나를 위한 `rune`형을 제공합니다.
- Rust는 `str`과 `String`을 UTF-8로 저장하며 Unicode 스칼라 값을 나타내는 `char`를 제공합니다.

프로그램 실행 중 문자열이 메모리에 놓이는 방식과 파일 또는 네트워크에서 전송되는 방식은 다를 수 있습니다. 파일과 네트워크 프로토콜에서는 높은 호환성과 공간 효율 덕분에 UTF-8을 널리 사용합니다.
