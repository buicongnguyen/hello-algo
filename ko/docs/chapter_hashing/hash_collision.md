# 해시 충돌

키 공간은 보통 버킷 수보다 크므로 충돌을 완전히 피할 수 없습니다. 대표적인 해결 방식은 **분리 연결법**과 **개방 주소법**입니다.

## 분리 연결법

각 버킷은 같은 해시 인덱스를 가진 키–값 쌍의 리스트를 저장합니다. 삽입과 삭제가 단순하고 버킷 수보다 많은 원소를 담을 수 있지만 연결 구조에 메모리가 추가로 들고 분산 접근은 캐시에 덜 친화적일 수 있습니다.

![분리 연결법을 사용하는 해시 테이블](hash_collision.assets/hash_table_chaining.png)

체인이 너무 길어지면 일부 구현은 최악의 성능을 개선하기 위해 버킷을 균형 트리로 바꿉니다.

## 개방 주소법

모든 키–값 쌍을 버킷 배열 안에 직접 저장합니다. 처음 위치가 이미 사용 중이면 정해진 규칙에 따라 다른 위치를 탐색합니다.

### 선형 탐사

키를 찾거나 빈 칸을 만날 때까지 다음 위치를 차례로 확인합니다. 단순하고 지역성이 좋지만 원소가 연속해서 뭉치는 군집이 생기기 쉽습니다.

![선형 탐사의 키 분포](hash_collision.assets/hash_table_linear_probing.png)

삭제할 때 칸을 즉시 빈 상태로 만들면 뒤의 키를 찾는 탐색이 너무 일찍 멈출 수 있습니다. 따라서 보통 **삭제됨**을 나타내는 특별한 tombstone 표식을 사용합니다.

![개방 주소법에서 삭제로 생기는 문제](hash_collision.assets/hash_table_open_addressing_deletion.png)

### 제곱 탐사

한 칸씩 이동하지 않고 $1^2, 2^2, 3^2, \ldots$의 오프셋을 시도합니다. 1차 군집은 줄이지만 처음 위치가 같은 키들이 같은 탐사 순서를 따를 수 있습니다.

### 이중 해싱

두 번째 해시 함수로 이동 간격을 정합니다. 간격과 테이블 용량을 알맞게 선택하면 전체 테이블을 방문하면서 군집을 줄일 수 있습니다.

## 프로그래밍 언어의 선택

- Python `dict`와 `set`은 최적화된 개방 주소법 변형을 사용합니다.
- Java `HashMap`은 분리 연결법을 쓰고 긴 버킷을 트리로 바꿀 수 있습니다.
- C++ `std::unordered_map`은 보통 버킷과 체이닝을 사용하지만 세부 사항은 구현에 따라 다릅니다.
- Rust `HashMap`은 최적화된 개방 주소 설계를 사용하고 해시 빌더를 선택할 수 있습니다.

모든 상황에 가장 좋은 전략은 없습니다. 키 유형, 접근 패턴, 메모리 제한, 보안 요구, 표준 라이브러리 특성에 맞춰 선택해야 합니다.
